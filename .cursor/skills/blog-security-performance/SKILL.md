# blog — セキュリティ・パフォーマンス

コードの追加・変更・レビューでは **セキュリティ** と **パフォーマンス** を必ず考慮する。

## いつ使うか

- 新規 API・ハンドラ・フロントコンポーネントを書くとき
- 既存コードの修正・リファクタ時
- プルリクや設計のレビュー時
- ユーザーが「セキュリティ」「パフォーマンス」「負荷」に言及したとき

## セキュリティ

- **認証・認可**: 管理者向けは X-Admin-Key または Bearer セッション。公開 API に機密を返さない。
- **入力**: 型・長さ・形式を検証。URL・ファイル名はスキーム・パスをホワイトリスト検討。
- **出力**: エラーに内部パス・スタックトレースを出さない。機密をログに書かない。
- **依存**: `go list -m -json all` / `npm audit` で既知脆弱性を確認。

## パフォーマンス

- **バックエンド**: N+1 を避ける。必要件数だけ取得。重い処理はキャッシュ・非同期を検討。
- **フロント**: 大リストは仮想スクロール・ページング。画像は LCP 要素は eager、それ以外は lazy を検討。
- **API**: レスポンスサイズ・クエリコストを意識。変更後は遅延・リソースを確認してから完了とする。

## チェック

実装・修正のたびに: 「認証抜け・入力の信頼・情報漏れはないか」「不要な重い処理・N+1・巨大レスポンスはないか」。
