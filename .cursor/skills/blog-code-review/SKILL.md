# blog — コードレビュー方針（Claude 設定準拠）

このスキルは `~/.claude/CLAUDE.md` の内容を blog モノレポ用に反映したもの。  
コードレビューを行うときは、**同僚のシニアエンジニアとして厳しく・具体的に**指摘すること。「問題なさそう」で済ませず、必ず以下のチェックリストに沿って確認する。

---

## 0. Must/Should の判断基準（運用ルール）

- **Must**: セキュリティ事故・データ破壊・権限漏れ・重大なパフォーマンス劣化・設計原則の明確な違反に繋がるもの
- **Should**: 将来的なバグ/運用コスト増に繋がる可能性が高いもの、読みやすさ・保守性の大きな改善
- **Nice to have**: 方向性の提案、将来の改善余地。いま直さなくても良いが有益なもの

迷った場合は Must に上げず、Should / Nice に落として理由を書く。

## 1. レイヤー境界（Clean Architecture / DDD）

- [ ] Domain 層にインフラ依存（DB・外部 API 等）が混入していないか
- [ ] UseCase / Application 層が HTTP や gRPC の詳細に直接依存していないか
- [ ] 依存の方向が **外→内** になっているか
- [ ] 具象型に依存せず、インターフェース越しに依存しているか

## 2. エラーハンドリング

- [ ] `err` を無視していないか（`_` で捨てていないか）
- [ ] エラーにコンテキストが付与されているか（例: `fmt.Errorf("読み込み失敗: %w", err)`）
- [ ] ドメインエラーとインフラエラーが適切に分離されているか
- [ ] エラーメッセージや内部エラーがそのままクライアントに露出していないか

## 3. 命名・可読性

- [ ] 変数名・関数名から意図が読み取れるか（`u` より `user` など）
- [ ] Go は `MixedCaps`、proto / SQL / TS はチームの命名規則に沿っているか
- [ ] コメントは「何をするか」ではなく **「なぜそうするか」** を説明しているか
- [ ] 関数が単一責任になっているか（長すぎないか、ネストが深すぎないか）

## 4. テスト

- [ ] ハッピーパスだけでなく、エラー系・境界値のテストがあるか
- [ ] Go はテーブル駆動テストで書かれているか
- [ ] モック・フェイクの使い方が適切か（外部依存を直接叩いていないか）
- [ ] テスト名から「何を検証しているか」が明確か

## 5. セキュリティ

- [ ] SQL インジェクションの可能性がないか（プレースホルダ使用か）
- [ ] トークン・パスワード等の機密情報をログに出していないか
- [ ] 認証・認可チェックが漏れていないか（管理 API / 管理画面は特に注意）
- [ ] 入力値のバリデーションがされているか（長さ・形式・必須チェックなど）
- [ ] 公開 API / 管理 API で返すエラーが内部情報（テーブル名・SQL・スタック）を含まないか
- [ ] 管理画面（`frontend/src/app/admin/**`）で例外の `message` をそのまま表示していないか
- [ ] セッションや AdminKey の扱いが漏れていないか（ログ・URL・画面出力）

## 6. パフォーマンス

- [ ] N+1 クエリが発生していないか
- [ ] ループ内で不要な DB / 外部 API 呼び出しをしていないか
- [ ] 不要なメモリアロケーションやコピーがないか
- [ ] インデックスが効くクエリか（WHERE 句・ORDER BY の列）
- [ ] Cloud Run / Cloud SQL のコストに直結する無駄（重いデプロイ手順・常時稼働・過剰な並列）がないか

## 7. 技術別の追加観点

### Go（言語仕様 / Style）

- **前提（一次ソース）**
  - [ ] 迷ったら [The Go Programming Language Specification](http://go.dev/ref/spec) と [Uber Go Style Guide](https://github.com/uber-go/guide) を参照する
  - [ ] 言語仕様に反する実装、または Uber Go Style に明確に反する実装は Must として指摘する
- **パッケージ / 構成**
  - [ ] `main` にビジネスロジックが混入していないか（wire・起動のみ）
  - [ ] 循環依存が発生していないか
- **命名・公開範囲**
  - [ ] export は最小限か（公開が必要なものだけ `UpperCamelCase`）
  - [ ] Getter に `Get` を付けていないか（`Name()` で良いか検討）
  - [ ] コンストラクタは `NewXxx` で統一されているか
- **エラーハンドリング**
  - [ ] `panic` を安易に使っていないか
  - [ ] `fmt.Errorf("context: %w", err)` でラップされているか
  - [ ] `errors.Is` / `errors.As` が適切に使われているか
- **context**
  - [ ] 外部 I/O（DB/HTTP/gRPC）は `ctx` を第一引数で受け取っているか
  - [ ] ライブラリ内で `context.Background()` / `TODO()` を新規に作っていないか（呼び出し元から受け取る）
  - [ ] `WithCancel` / `WithTimeout` の `cancel` を `defer cancel()` しているか
- **goroutine / 並行**
  - [ ] goroutine リークの可能性がないか（終了待ち、cancel、close）
  - [ ] 必要に応じて `sync.WaitGroup` / `errgroup.Group` を使っているか
- **インターフェース設計**
  - [ ] interface は利用側パッケージに定義されているか
  - [ ] fat interface になっていないか（必要最小限か）
- **データ構造・パフォーマンス**
  - [ ] スライスの capacity を意識しているか（`make([]T, 0, n)`）
  - [ ] map のゼロ値と `make` の違いを理解した使い方か
  - [ ] ポインタ/値の選択が妥当か（可変性・コピーコスト）
- **テスト**
  - [ ] テーブル駆動で整理されているか
  - [ ] 並行処理を含むなら `-race` でも安全か（少なくとも意識されているか）

### gRPC / Protobuf（connect-go を含む）

- `context.Context` が適切に引き回されているか（タイムアウト / cancel 忘れがないか）
- interface は **利用側** に定義されているか
- Protobuf:
  - フィールド名・番号が既存と衝突していないか
  - ステータスコードが適切か（`InvalidArgument` / `NotFound` / `FailedPrecondition` 等）
- Connect:
  - [ ] 公開 API で内部エラーをそのまま返していないか（`errors.go` 等のマッピング方針に沿う）
  - [ ] Admin API は Bearer セッション or AdminKey 必須になっているか

### SQL / migration

- トランザクション境界が適切か（必要以上に広く/狭くなっていないか）
- NULL 許容カラムの扱いが適切か（Go 側の型との対応）
- マイグレーションがロールバック可能か（down を想定した設計になっているか）

### TypeScript / Next.js フロントエンド

- `any` を安易に使っていないか（型をきちんと定義しているか）
- コンポーネントの責務が適切に分割されているか（巨大コンポーネント化していないか）
- `useEffect` の依存配列が正しいか（無限ループや取りこぼしがないか）
- XSS / CSRF への配慮がされているか（`dangerouslySetInnerHTML` / フォーム送信など）
- 不要な再レンダリングが発生していないか（メモ化や依存管理）

## 8. レビュー時の出力フォーマット

指摘を書くときは、必ず以下のフォーマットで出力する。

```md
## 🔴 Must（必ず修正）
- [path/to/file.go:123] 問題の説明 → 修正案

## 🟡 Should（できれば修正）
- [path/to/file.tsx:45] 問題の説明 → 修正案

## 🟢 Nice to have（提案）
- [path/to/file.sql:10] 提案内容
```

### 補足

- 「問題なし」で終わらせず、**観点ごとに確認した結果**を簡潔にコメントする。
- 迷った場合は Must にせず Should / Nice に落とすが、理由も書く。
