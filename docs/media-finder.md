# メディアファインダー（将来実装）設計

一度アップロードした画像を一覧で選び、サムネイルや本文に再利用できるようにするための**将来実装用**の設計・要件をまとめたドキュメント。post-thumbnail-and-media の Phase 2（アップローダー）の次の拡張として位置づける。

---

## 1. 目的・概要

- **目的**: アップロード済みメディアを一覧・検索し、記事編集画面から「既存の画像を選択」してサムネイルや本文に挿入できるようにする。
- **現状**: 「都度アップロード → URL を返す」のみで、一覧表示やメタデータの管理はない。メディアファインダー導入により、同じ画像を複数記事で再利用したり、過去にアップロードした画像を探して挿入したりできるようになる。

---

## 2. 要件

- 管理者が**メディアライブラリ（一覧）**を閲覧できる。
- 一覧から画像を選択し、その URL を**サムネイル欄**や**本文**に挿入できる（再利用）。
- オプション: キーワード・日付・MIME タイプでの絞り込み、ページング。

---

## 3. データモデル案

### 案 A（推奨）: メディア用テーブルを新設する

- **テーブル例**: `media`（`id`, `storage_key`, `url`, `content_type`, `size_bytes`, `uploaded_at`, `alt_or_name` など）。
- アップロード時にレコードを作成し、一覧 API はこのテーブルを参照する。
- **メリット**: メタデータ・検索・ページングがしやすい。alt や名前を DB で管理できる。
- 既存の `POST /upload` は「ファイル保存 + media レコード作成」に拡張するか、別エンドポイントで「アップロード＆登録」にまとめる。

### 案 B: DB は持たず、ストレージの ListObjects のみで一覧する

- ストレージ（GCS / R2）の ListObjects API で一覧を取得する。
- **メリット**: 実装が簡単。DB マイグレーション不要。
- **デメリット**: メタデータ（alt など）や柔軟な検索がしづらい。ページングはストレージ API の仕様に依存する。

**本設計では案 A を推奨**とする。案 B はプロトタイプや小規模運用向けの選択肢として記載する。

---

## 4. API 案

- **ListMedias**: 管理者認証必須。ページング・オプションでフィルタ（content_type、日付範囲など）。レスポンス: メディアの id, url, content_type, uploaded_at, alt_or_name など。
- **既存の POST /upload**: 「ストレージに保存して URL を返す」のまま維持する。「メディアとして登録」は別途、次のいずれかで対応する。
  - アップロード後に media レコードを挿入する（同一リクエスト内で保存＋登録）。
  - または Connect RPC の `RegisterMedia` のようなエンドポイントを用意し、アップロード済み URL をメディアライブラリに登録する。
- **取得**: **GetMedia**（単体）は任意。一覧から選ぶだけなら List で十分とする。詳細表示・編集用に Get を用意するかは実装時の検討事項とする。

---

## 5. フロント（管理画面）

- **メディアライブラリ画面**: 一覧表示（サムネイルグリッドまたはリスト）、ページング、絞り込み。クリックで「この URL を使う」とし、記事編集画面に戻る、またはモーダルで選択して URL を渡す。
- **記事編集・新規作成**:
  - 「サムネイル」欄に**「既存から選択」**ボタンを追加。メディアライブラリ（またはモーダル）を開き、選択した画像の URL をサムネイル欄にセットする。
  - 本文用に「画像を挿入」で**「既存から選択」**を用意する。選択した画像の URL を `![alt](url)` 形式でカーソル位置に挿入する。
- 既存の「ファイルを選択してアップロード」は残し、**「新規アップロード」**と**「既存から選択」**の両方を選べるようにする。

---

## 6. 既存アップロードとの共存

- 現状の「アップロード → URL だけ返す」は、メディアテーブルを持たない限り「登録」されない。
- メディアファインダー導入後は、次のいずれかを選ぶ:
  1. **アップロード時に同時に media レコードを作成する**: POST /upload 成功後に media テーブルに 1 件挿入。一覧に自動で載る。
  2. **「アップロード」と「メディアとして登録」を分離する**: アップロードは URL 返却のみ。別途「メディアライブラリに追加」操作で登録する。
- 既に DB に保存されている `posts.thumbnail_url` や本文中の URL は、media テーブルに履歴がなくても**表示は可能**。ファインダーは「これから登録するメディア」の一覧として使い、過去の URL はそのまま参照できる。

---

## 7. 実装フェーズ・優先度

- 本機能は **post-thumbnail-and-media の Phase 2 の後**の拡張（Phase 2.5 または別フェーズ）とする。
- R2 実装とは独立して、次の順で実装する想定:
  1. メディアテーブルの追加（マイグレーション）
  2. ListMedias API（および必要に応じて GetMedia）
  3. POST /upload との連携（アップロード時の media レコード作成、または RegisterMedia）
  4. 管理画面の「メディア」ページ（メディアライブラリ一覧）
  5. 記事編集・新規作成画面の「既存から選択」UI（サムネイル・本文）
